# Diario de Desarrollo — Habitly

## Autor
Iriome

## Proyecto
Habitly

## Tipo de documento
Registro de avance / diario de desarrollo

## Objetivo
Este documento recoge, de forma cronológica y resumida, las decisiones, avances y cambios realizados durante el desarrollo del proyecto Habitly.  
Su finalidad es dejar constancia del proceso de evolución del sistema, facilitar la trazabilidad del trabajo realizado y servir como apoyo tanto para la memoria principal como para futuras versiones del proyecto.

---

# Entrada 01 — Día anterior

## Estado general del proyecto al inicio
El proyecto se encontraba en una fase de transición importante.  
Hasta ese momento, la base funcional del sistema había sido desarrollada como una **fase BETA** utilizando **BlueJ** como entorno de trabajo, con una arquitectura orientada a objetos sencilla pero funcional.

La BETA ya incluía:

- una superclase `Vivienda`
- dos subclases (`Piso` y `Casa`)
- un menú funcional por consola
- registro de viviendas
- listado de inventario
- registro de pagos
- aplicación de subida anual / IPC
- control de errores con `try-catch`
- almacenamiento en `ArrayList<Vivienda>`

A partir de ese punto, se decidió comenzar la reorganización del proyecto para transformarlo en una versión más seria y escalable.

---

## Decisiones importantes tomadas

### 1. Cambio definitivo de nombre del proyecto
Se decidió cambiar el nombre del proyecto de su nombre anterior a:

# Habitly

Este nombre pasa a ser el nombre oficial del software y debe utilizarse a partir de ahora en:

- documentación
- memoria
- anexos
- roadmap
- prompts
- repositorios
- desarrollo técnico

---

### 2. Cierre oficial de la fase BETA
Se dejó claro que la **fase BETA se considera terminada**.

Esto implica que:

- la BETA ya no es la fase activa del proyecto
- la BETA pasa a ser la base inicial cerrada
- la **versión 1.0 pasa a ser la nueva fase activa de desarrollo**

Este cambio también afecta a toda la documentación del proyecto, ya que a partir de ahora la memoria principal debe reflejar que la BETA es una etapa finalizada y no la versión actual.

---

### 3. Replanteamiento completo de la documentación
Se tomó la decisión de **rehacer la documentación del proyecto desde una base más seria y organizada**, con el objetivo de que Habitly quede correctamente estructurado también a nivel documental.

Se definió una organización más clara basada en:

- una **Memoria Principal** como documento madre
- varios **anexos de evolución por versión**
- un **roadmap específico para la v1.0**
- y prompts de apoyo para desarrollo con IA

Esto supone un cambio importante respecto a una documentación más improvisada o centrada solo en la BETA.

---

### 4. Decisión sobre la estructura documental futura
Se estableció que las futuras versiones del proyecto se documentarán de forma separada mediante anexos independientes.

Formato acordado:

- `Anexo_01_Version_1.0_Habitly`
- `Anexo_02_Version_X.Y_Habitly`
- etc.

Esto permitirá que la memoria principal no se sobrecargue y que cada evolución del sistema quede bien registrada de forma independiente.

---

## Desarrollo documental realizado

### 5. Elaboración de una nueva Memoria Principal
Se comenzó a reconstruir la **memoria principal de Habitly** desde una perspectiva mucho más formal, seria y pensada para albergar todo el proyecto, no solo la BETA.

Durante este proceso se definió que la memoria principal debía:

- presentar Habitly como proyecto global
- explicar el origen del sistema
- recoger la base conceptual
- describir la arquitectura inicial
- y servir como documento madre para todas las fases

También se empezó a trabajar el enfoque de estilo y presentación del documento.

---

### 6. Definición del estilo de maquetación del documento
Se concretaron decisiones prácticas para el formato de la documentación en Writer, incluyendo recomendaciones sobre:

- títulos
- subtítulos
- estructura de páginas
- uso de saltos de página
- jerarquía visual
- estilo de memoria académica / técnica real

Esto se hizo con la intención de que la documentación de Habitly no sea solo correcta en contenido, sino también limpia y defendible en presentación.

---

### 7. Creación del Anexo 01 de la versión 1.0
Se desarrolló un primer gran documento específico para la nueva etapa del proyecto:

# `Anexo_01_Version_1.0_Habitly`

Este anexo se planteó como el documento encargado de explicar:

- el paso de la BETA a la v1.0
- el nuevo enfoque del proyecto
- la ampliación funcional
- la evolución técnica
- y la dirección general del desarrollo

En ese momento, el documento todavía fue afinado posteriormente, pero ya quedó establecida su función principal dentro del proyecto.

---

### 8. Creación del roadmap inicial de la v1.0
También se comenzó a diseñar el **roadmap de desarrollo de Habitly v1.0**, con el objetivo de organizar mejor la evolución del proyecto.

Se definió que este roadmap debía servir para:

- ordenar el trabajo
- evitar crecer sin control
- establecer prioridades
- y orientar la implementación real

Esto fue clave porque el proyecto dejó de estar en fase “exploratoria” y empezó a necesitar una planificación más seria.

---

## Enfoque funcional decidido para la nueva etapa

### 9. Nueva visión del sistema: más allá de la vivienda
Se definió que Habitly ya no debía verse solo como un sistema para “guardar casas o pisos”, sino como un software que empieza a representar una lógica inmobiliaria más rica.

Esto implica que el sistema comenzará a considerar también:

- usuarios
- relaciones entre usuarios y viviendas
- pagos
- gastos
- rentabilidad
- control económico básico

Esta decisión marca un cambio importante respecto a la BETA, que estaba mucho más centrada en la vivienda como objeto aislado.

---

### 10. Definición de doble perspectiva de usuario
Se acordó que la versión 1.0 del proyecto debía empezar a contemplar dos perfiles funcionales importantes:

#### a) Usuario arrendatario / residente
Perfil pensado para representar a la persona que alquila o habita una vivienda.

#### b) Usuario arrendador / propietario-gestor
Perfil pensado para representar a quien posee una o varias propiedades y las alquila o gestiona.

Esta decisión amplía considerablemente el alcance del sistema y servirá como base para futuras ampliaciones funcionales.

---

### 11. Primeras ideas de lógica económica futura
Se empezó a consolidar la idea de que Habitly deberá ir incorporando una lógica más realista para el control de ingresos y gastos.

Entre los conceptos mencionados y considerados como relevantes para la evolución del sistema se encuentran:

- beneficio total
- gastos por vivienda
- seguimiento económico
- análisis de rentabilidad
- información más útil para arrendadores

Esto todavía no supone implementación directa, pero sí una línea clara de diseño para la v1.0.

---

# Entrada 02 — Día actual

## Estado general al comenzar el día
El proyecto ya contaba con una base documental inicial más clara, pero todavía era necesario:

- ajustar mejor el Anexo 01
- afinar el roadmap
- definir mejor el punto de partida real de la v1.0
- y preparar un contexto sólido para poder trabajar con otra IA en el desarrollo técnico

Además, se consolidó la idea de abandonar definitivamente BlueJ como entorno principal de desarrollo.

---

## Decisiones técnicas importantes

### 12. Migración conceptual del proyecto a IntelliJ IDEA
Se decidió que la **versión 1.0 de Habitly se desarrollará ya directamente en IntelliJ IDEA**, dejando BlueJ como entorno correspondiente únicamente a la fase BETA.

Este cambio tiene varias implicaciones importantes:

- mejora de organización del proyecto
- mejor preparación para Git
- estructura más profesional
- mayor facilidad de refactorización
- entorno más adecuado para una evolución seria del sistema

Esto marca una transición técnica muy importante en el ciclo de vida del proyecto.

---

### 13. Integración futura con Git y GitHub
Se dejó establecido que a partir de esta nueva fase, el proyecto trabajará ya con:

- **Git**
- **GitHub**

Esto convierte a Habitly en un proyecto con una metodología de desarrollo más profesional y facilita:

- control de cambios
- organización de versiones
- trazabilidad
- evolución técnica más limpia

También se reforzó la idea de que el proyecto tendrá identidad propia como software serio y no solo como práctica aislada.

---

## Revisión y adaptación de documentación

### 14. Revisión crítica del Anexo 01
Se detectó que una versión del **Anexo 01** había sido mejorada formalmente, pero quizás se había alejado demasiado del estado real del proyecto.

Se hizo una reflexión importante sobre la diferencia entre:

- un documento “bonito” o muy ambicioso
- y un documento realmente útil y fiel al estado actual de Habitly

A partir de ahí, se decidió **no rehacer desde cero**, sino:

- tomar como base la versión del Anexo 01 que ya había sido aprobada
- y adaptarla con las nuevas decisiones tomadas

Esta decisión fue importante para mantener coherencia documental sin perder realismo.

---

### 15. Adaptación definitiva del Anexo 01 a la situación real del proyecto
Se rehizo el **Anexo_01_Version_1.0_Habitly** tomando como base la versión previamente validada, pero añadiendo y ajustando:

- que la BETA ya está terminada
- que la v1.0 es la fase activa
- que el proyecto pasa a IntelliJ IDEA
- que se incorporará Git/GitHub
- que se mantiene el enfoque canario
- que el sistema usará IGIC en esta etapa
- que la v1.0 incorporará doble perspectiva de usuario
- que se contemplarán más atributos y gastos por vivienda

Este ajuste permitió que el anexo quedara mejor alineado con la realidad actual del proyecto.

---

### 16. Adaptación del roadmap a la nueva situación real
Posteriormente, se hizo el mismo trabajo con el roadmap de la v1.0.

Se revisó y reescribió para que reflejara correctamente:

- la migración a IntelliJ
- la consolidación técnica inicial
- la revisión de arquitectura
- la ampliación del modelo de vivienda
- la futura incorporación de perfiles de usuario
- la evolución de la lógica económica
- y el cierre realista de la v1.0

Esto dejó el roadmap mucho más coherente con el estado actual de desarrollo.

---

## Consolidación conceptual del producto

### 17. Confirmación del enfoque canario del sistema
Se dejó establecido de forma clara que Habitly, en la BETA y en la v1.0, estará orientado principalmente al contexto de **Canarias**.

Esto significa que el sistema trabajará fiscalmente con:

# IGIC

y no con IVA en esta fase.

También se dejó definida una idea importante de evolución futura:

- la futura **v1.1** o versiones posteriores podrían adaptarse a la península
- esa futura ampliación incluiría IVA y una lógica fiscal más general para España

Pero se decidió conscientemente que **eso todavía no toca en la v1.0**.

Esta decisión ayuda mucho a mantener el proyecto controlado y bien enfocado.

---

### 18. Confirmación de los límites de la v1.0
También se consolidó algo muy importante para evitar errores de desarrollo:

## Qué NO conviene hacer todavía

Se reafirmó que en esta etapa todavía no se quiere meter:

- interfaz gráfica
- base de datos real
- persistencia compleja
- autenticación avanzada
- sobrearquitectura innecesaria

Y se reforzó la filosofía de crecimiento del proyecto basada en:

- hacer primero una base sólida
- ampliar después con criterio
- no sobredimensionar antes de tiempo

Esto será muy importante para mantener Habitly coherente y sostenible.

---

## Trabajo clave realizado hoy

### 19. Creación del Prompt Maestro Definitivo de Habitly v1.0
Uno de los avances más importantes del día fue la creación de un **Prompt Maestro Definitivo** para poder trabajar con otra IA en el desarrollo del proyecto sin tener que volver a explicarle todo desde cero.

Este prompt fue diseñado para transmitir de forma completa y estructurada:

- el contexto del proyecto
- la historia de la BETA
- el paso a la v1.0
- el cambio a IntelliJ
- el uso de Git/GitHub
- el enfoque canario
- el uso de IGIC
- la doble perspectiva de usuario
- la filosofía de desarrollo
- las reglas sobre cuándo dar código y cuándo no
- y la forma correcta de ayudar en Habitly

Este prompt se convierte en una herramienta clave para el desarrollo asistido del proyecto en adelante.

---

### 20. Definición del rol ideal de la IA de apoyo
Dentro de ese mismo trabajo, también se dejó claro qué papel debe cumplir la IA cuando se use para ayudar en el proyecto.

Se definió que la IA no debe actuar solo como “generadora de código”, sino como una combinación de:

- arquitecto de software
- mentor técnico
- analista funcional
- asistente de desarrollo Java

Esto es importante porque orienta el uso de IA hacia un modelo más inteligente y más útil para el crecimiento real del proyecto.

---

Al finalizar la jornada, Habitly ha quedado en una situación mucho más madura y clara que al inicio.

Actualmente, el proyecto dispone ya de:

- una identidad definitiva como **Habitly**
- una **BETA cerrada y finalizada**
- una **v1.0 definida como fase activa**
- una base funcional inicial ya construida
- una dirección documental mucho más clara
- un **Anexo 01 actualizado**
- un **Roadmap actualizado**
- un **Prompt Maestro Definitivo**
- una futura transición de desarrollo a **IntelliJ + Git + GitHub**
- y una visión bastante más sólida de hacia dónde debe crecer el sistema

En este punto, el proyecto ya está mucho mejor preparado para comenzar la implementación seria de la versión 1.0.
